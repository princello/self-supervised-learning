# Part 3 lab — base model report

Offline validation for "The Machine That Learns to Answer" (SFT explainer).
Everything here is measured, seeded, and reproducible; nothing is rounded up.

## What was built

- **Micro-world corpus** (`corpus.txt`, generated by `gen_corpus.py`, seed 0):
  30 creatures, each with a colour, a home, and a food (90 facts). Facts appear
  3x each across 6 templates (canonical `is / lives in the / eats` twice,
  variant `looks / rests in the / likes` once). Interleaved "quiz pages": all
  90 questions (`what colour is the fox?` / `where does the mole live?` /
  `what does the owl eat?`) in shuffled blocks of 5, **never followed by
  answers**. No `q:`/`a:` markers anywhere in the base corpus.
- **Base model** (`model.py`, `train_base.py`): char embedding (31 x 8) ->
  concat last K=26 chars (208) -> dense tanh hidden (96) -> softmax(31).
  **23,319 params.** Trained with plain SGD + momentum(0.9) — the exact loop
  the browser JS will re-implement. No attention, no layernorm, no Adam.

### Two findings the page build must know

1. **Fresh-start augmentation is required.** With plain next-char training the
   model never sees a context that *starts* at a sentence (training windows
   always contain full preceding corpus text), so bare prompts like
   `the fox is ` produced garbage (3.3% fact completion). Fix in
   `train_base.py`: with p=0.5, a training example's context is truncated at
   the current sentence start (earlier chars replaced by newline padding) —
   exactly what a demo-time prompt looks like. The **browser SFT loop does not
   need this**; it only affected base training (done offline in Python).
2. **Batch size mattered more than steps.** batch 128 plateaued at 84.4% facts
   (loss 0.272); batch 256 with identical step count reached 100% (loss
   0.01402). Gradient noise, not capacity, was the blocker.

### 4. SFT smoke test — PASS (pipeline works end to end)

16 demonstrations in format `q: <question>\na: <answer>\n` (392 answer-line
positions; loss on the answer line only, including the `a: ` marker so the
model learns to emit it). **300 steps x batch 24, lr 0.02, momentum 0.9**:

- exact-answer rate before SFT: train 0/16, held-out 0/8
- exact-answer rate after SFT: **train 16/16 = 100%**, held-out 0/8
- answer-line loss: 2.5962 -> 0.0056
- compute: **4.96e8 MACs** (steps x batch x 3 x 22,944), within the 5e8 budget

Per-seed spread (same config, SFT seeds 0/1/2): train **16/16, 16/16, 16/16**;
held-out **0/8, 0/8, 0/8**; final loss 0.0056 / 0.0008 / 0.0160.

**lr warning:** lr 0.15 catastrophically garbles the base model (1/16 train);
lr 0.08 -> 13/16, lr 0.05 -> 14/16, **lr 0.02 -> 16/16**. Stay ~0.02.

**Honest negative result — no generalization to un-demoed facts, in ANY
configuration tried.** Exact-answer rate on held-out questions is 0% for:
16 demos/300 steps (0/8); 60 demos/2,000 and 4,000 steps (0/30, 6.6e9 MACs,
offline-only budget); answer-line fresh-start augmentation (0/30); and an
alternative base trained with prefix-scramble augmentation (0/30, below).
Held-out outputs are fluent but wrong ("what colour is the jay?" ->
"a: the bat is black."). Mechanism (measured): answering an un-demoed
question requires *copying* the entity name from the question prefix into the
answer, and a fixed-window MLP with no attention must learn position-specific
copy mappings — 16-60 demos cannot teach that. Base fact recall itself only
partially survives the demo geometry: greedy `a: the <x> is ` -> 23/30, but
`<question>\na: the <x> is ` -> 5/30 (fresh base).

We tested a base-training fix — *scramble augmentation* (replace pre-sentence
context with the tail of a random boundary-aligned corpus window; code path
ships in `train_base.py`, `P_SCRAMBLE`). It fixes teacher-forced prefix
robustness (question+`a: ` recall 26-30/30) but does NOT unlock held-out QA
(0/30 — the copy step is still the blocker), and it costs SFT convergence
(best 15/16 in budget vs 16/16). So the shipping base uses `P_SCRAMBLE=0`.
Full numbers: `results/heldout_diagnostics.json`.

**Consequence for the page (blueprint section 5, "held-out animals"):** the
planned bet "can it answer about an animal it never practiced on?" resolves
to **NO at this scale** — which is still a great, honest reveal (toy scale
lacks the machinery for the LIMA effect; contrast with real-scale SFT in
prose), but the section cannot show a positive held-out rate. The live-demo
quiz must use demonstrated facts. The hero moment is unaffected: before SFT a
question begets another question; after SFT it begets its answer.

## Deviations from the brief (all deliberate, documented)

- **`Q:`/`A:` -> lowercase `q:`/`a:`.** Uppercase letters are not in the
  31-char vocab (adding `Q`,`A` would push it to 33 > 32). The colon *is* in
  the vocab, unused by the base corpus, so `q: `/`a: ` gives the same
  "markers arrive only at SFT time" story. Demo agents must use lowercase.
- Params 23,319 < ~30k cap, but note the *real* ceiling is the 65 KB b64
  export: params must stay <= ~24.4k. H=96 is at that ceiling.
- `z` and (lowercase) `q` never occur in the base corpus; `q` first appears at
  SFT time in `q: ` (embedding trained then — works, see item 4).

## Recommended SFT starting points for the demo agents

- optimizer: SGD + momentum 0.9, **lr 0.02** (0.05 works but drops to 14/16;
  >= 0.08 visibly damages the base model)
- 16 demos, **300 steps x batch 24** = 4.96e8 MACs =~ 2-3 s of JS. Loss on
  answer-line chars only (include the `a: ` marker chars in the loss).
- quiz only demonstrated facts; keep a "before" model handy for the
  question-continues-question contrast
- sampling: temperature 0.8 for the hero demo (0/200 accidental answers),
  greedy for post-SFT answers
- fp16-decoded weights are behaviourally identical to fp32 here (90/90)
